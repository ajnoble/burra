Phase 7: Stripe Connect Architecture

How should the payment flow be structured?

A

Server Actions + API Routes (Recommended)

Keep the existing pattern. Server actions for mutations (create checkout session, handle onboarding), API route for the webhook endpoint. Stripe logic in a shared lib/stripe.ts helper.

Flow:
Member Dashboard → "Pay Now" → Server Action creates Checkout Session → Redirect to Stripe → Stripe sends webhook → API Route updates transaction

Pros

  • Matches existing codebase patterns (server actions everywhere)
  • Type-safe end-to-end with Drizzle
  • Webhook is naturally an API route (not a server action)
  • Minimal new abstractions

Cons

  • Stripe logic split between actions/ and api/ directories
B

Dedicated Stripe Service Layer

Create a services/stripe/ module that encapsulates all Stripe interactions. Server actions and API routes call into the service layer.

Flow:
Server Action → StripeService.createCheckoutSession() → Stripe API
Webhook API Route → StripeService.handleEvent() → DB update

Pros

  • All Stripe logic in one place
  • Easier to test Stripe interactions in isolation

Cons

  • New pattern not used elsewhere in codebase
  • Extra abstraction layer for ~3-4 Stripe calls
  • YAGNI — premature for this phase's scope
C

All API Routes

Move payment-related logic to API routes instead of server actions. Webhook, checkout creation, and onboarding all live under app/api/stripe/.

Flow:
Client fetch → POST /api/stripe/checkout → Stripe API
Stripe → POST /api/stripe/webhook → DB update

Pros

  • All Stripe code co-located under api/stripe/
  • Standard REST approach

Cons

  • Breaks from server action pattern used everywhere else
  • Loses server action benefits (type safety, progressive enhancement)
  • Client needs manual fetch calls instead of direct action imports
